AC_1: Architecture Coverage with on youtube free tier

In the context of Anypoint Platform, there is a runtime plane and control plane. Products and features that let you create, manage and discover mule applications without local setup are included in control plane like Design Center, Exchange, Management Center. Runtime plane is mule runtime along with services like connectors, extensions. Anypoint Studio includes the non-licensed / non-clustered setup of mule runtime in the studio for developers.

In the On-perm model, the control plane is hosted with Mulesoft for the customer org, mule agent connects mule applications running on VMs in customer datacenter to the control plane. In this setup, there are several mule applications run in a single JVM / mule server.

Cloud model is when both control and runtime plane hosted with Mulesoft with an iPaas cloudhub offering which is built on AWS Infrastructure as a Service offering. Mulesoft Cloudhub Offering provides different options like Cloudhub 1.0 (Cloud Workers), Cloudhub 2.0 (Cloud Containers), Runtime Fabric (On-perm containers).

In the migration journeys, there is a hybrid deployment model when the mule applications are deployed both in customer data center and in cloudhub.

Mule Error Handling_POV_1

Mulesoft Teams have enabled muleys in implementation projects with options when it comes to Error Handling with streamlined approach to Mule 4 Error Handling with core components of On Error Continue, On Error Propagate, Error Handler, Raise Error. Error Handling is enabled with Mule Default Error Handler, Application Level Error Handlers, Flow Level Error Handlers, Processor Level Error Handlers. One should go through scenarios explained based on calling flow (parent) and called flow (child flow) in the Mule Fundamentals course and Lightboard series videos.

Approach:Search related topics, follow Avatar Architects and understand their content.
  • Architects sugget to broadly understand application classification as monoliths and distributed. Monoliths are single deployable unit whereas distributed are multiple deployable units in application context. Monoliths implementations are somewhat simple, less expensive as compared to distributed because of complexity in cross cutting concern and how the application is designed for architecture characterstics (eg: scalability, availability etc).
  • Architects favor parallel workstream for devOps and development in implementation programs for fast release cycles and require technical teams to increase technical breadth in relevant areas that may not fall into the core skills.

Mule Error Handling_POV_2

Generally in projects, there are common error handlers which are developed and maintained as a common mule artifact (usually as a separate repository) and included as a dependency. We also rely on tracing transaction logs in ELK stacks along with kibana dashboards when the error crosses transaction boundaries and propagates across mule services in separate layers. We also persist some business errors that occur at application layer in middle-tier DB tables or with customizations in saas business applications like salesforce. We also rely on notifications by configuring alerts to manage outage and other error situations. In POCs, I took a simplified approach of creating common utility mule configuration file which can be copied in different mule applications.
