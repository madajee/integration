INTEGRATION 100 - Evolve

Evolving Game

In my experience, integrating systems, people and processes has been a hard ball, and the game of it just keeps evolving. There are several dimensions around this term, but my experience has been in system integration in the enterprise ecosystem. However with cloud, mobility and social integration ways, we all have to just keep learning, and relaxation is necessary exercise.

Integration Timeline

  • File Based Batch Integrations Sevetys going into Eightys

    Gained popularity by pushing data feeds into giant monoliths like mainframes and even used today with FTP on B2B integrations. Architects approched integrations with Batch Processes that consume and process data files.

  • Two Tier Architectural Style, MOMs, Real-Time Synchronous RPC Eightys going into Nintys

    Application Architecture was about 2-tiered architectural style developed with data residing in Oracle DB and Microsoft SQL Server, development of MOMs products like IBM MQ Series and ERPs like SAP, Oracle, WMS, Financial and Accounting Systems. Architects approached integration design with RPC style calls with shared databases and broker based CORBA architectures.

  • ERPs, Web 1.0, Multi-Tier Architecture with Sun JAVA, Microsoft .NET Stacks Nintys going to 2000

    Deployment of Packaged Applications like SAP, Oracle, JD Edwards, PeopleSoft gained momemtum and became system of records. Internet and Web based architectural style gained popularity with Java & .Net stack. Architects included asynchronous style message oriented middleware implemented with standards like MQ and JMS.

  • SOAP Webservices, EAI, Hub & Spoke Brokers 2000 to 2005

    Both business and technical leaders were required to integrate application silos and synchronous style converged towards SOAP based webservices and asynchronous style converged towards Brokers, Enterprise Application Integration (EAI) to address integration needs and challenges. Business use cases were orchestrated and composed with Business Process Manager(BPM) products.

    Niche integration vendors like WebMethods and TIBCO created products that implemented EAI and BPM patterns.Large enterprises and internet companies started enabling SOA architectures with interoperable services using the SOAP protocol and data exchange with XML, and deployments on Application Servers like Weblogic, Websphere based on Java EE6.

  • Enterprise Service Bus, Business Process Management 2005 to 2010

    Bus pattern started to emerge where architects analyzed that a lot of application and business logic is being developed in the middle tier and though it provides speedy services, we end up with criss cross connections and overtime, middleware becomes monoliths which are difficult to maintain and we have to correlate across application & transaction boundaries. In spite of centralized middleware, we will be better off with a service bus that is based on components, and uses DSL for rapid development. Application/Services connects with the Service Bus on edges and integration developers leverages service bus capabilities to address integration design tasks like validation, enrichment, transformation, routing. ESB products are usually runtimes with low memory footprints which are either provisioned along with Application Server like OSB on Weblogic server, or developed with open source community way like Mule ESB, Apache Camel with enterprise licensing backed by organizations like Mulesoft, Red Hat.

  • Cloud Computing,Mobility, Social, IAAS, PAAS , SAAS 2010 going to 2015

    We know about hypervisor, virtual servers but it is very different when you can provision a server with a click and choose the hardware configuration with a template. Cloud computing basically automated hardware provisioning with software tools and implementation of elastic design moved architecture layers into the cloud (IAAS, PAAS and SAAS). Products developed with cloud native architecture saw huge growth and all of these products are backed by APIs and platforms.

    Web interestingly is not the only client after the invention of the iPhone, and it is very different when there are platforms connecting millions of people. Smartphones are also enabling smart devices and there are billions of devices connected on the network which is called Internet of Things.

  • Microservices, API 2015 going to 2020

    Implementation stack at platform companies like Netflix explained that instead of monoliths, we have to develop components and services modeled around business domains that can be scaled and deployed independently. Architects approached integration with REST based API.
